## Issues found

**B1: `pnpm test` exits 1 — 8 BodyViewer test failures**

`src/components/__tests__/BodyViewer.test.tsx` has 8 failing tests. The test file was committed in spec 19 (commit `3522e32`) with prop `measurementId: string`, but the current `BodyViewer.tsx` on disk has a completely different interface: `measurements: Measurements | null, gender: BodyGender, onGenderChange, className`. The tests cannot find `data-testid="body-viewer-loading"`, cannot find buttons, and cannot confirm fetch calls because the prop rename (`measurementId` → `measurements`) changed how the component fetches.

Spec 20 did not introduce this regression — it was present in the tree before spec 20 ran. But the acceptance criterion explicitly states `pnpm test exits 0 (all existing tests pass)`, which is not met.

Ownership is ambiguous (spec 19 review should have caught it). However the criterion blocks merge.

Required fix: Update `BodyViewer.test.tsx` to use the current `measurements` / `gender` prop interface, or update `BodyViewer.tsx` to restore the `measurementId` prop.

**B2: `tsc --noEmit` has 15 type errors in `BodyViewer.test.tsx`**

All errors are `Property 'measurementId' does not exist on type 'IntrinsicAttributes & BodyViewerProps'`. Same root cause as B1. TypeScript strict mode is on; these are genuine errors in committed code (the test file is tracked).

### Non-blocking

**N1: Duplicate type definitions with divergent precision — `wizard.ts` vs `api.ts`**

`frontend/src/lib/api.ts:131`: `cascade_type: "fba" | "swayback" | "none"` (narrow literal union)
`frontend/src/store/wizard.ts:20`: `cascade_type: string` (widened)

Spec states "the DiagnosisResponse type here matches the wizard store's DiagnosisResult shape — they are the same data." They are not the same type. Assignment works (TypeScript widens silently) but the store loses the literal union, meaning downstream code that switches on `cascade_type` won't get exhaustiveness checking.

Suggested fix: either re-export `DiagnosisResponse` from `api.ts` and use it as the store's `DiagnosisResult`, or narrow `cascade_type` in `wizard.ts` to match.

**N2: `wizard.ts` modification beyond spec note boundary**

The spec notes say "Do not touch `wizard.ts` (currentStepIndex)" and the implementation notes acknowledge the wizard store was expanded as a necessary scaffold change. The expansion (adding `gradedPatternId`, `photoIds`, `diagnosisResult`, `cascadeScript`, `activeTool`) is required by the panels and makes the acceptance criteria achievable. This is a reasonable deviation, acknowledged in implementation notes. Flagged for Steph's awareness, not a blocker.

**N3: `DiagnosisPanel` auto-runs on mount with `eslint-disable-next-line react-hooks/exhaustive-deps`**

The exhaustive-deps disable is justified (mount-only by design; Zustand selectors are stable refs). The comment is present. However, the dependency array is empty `[]` but the effect closure captures `diagnosisResult`, `measurementsResponse`, `photoIds`, and `runDiagnosis`. If any of these change after mount, the auto-run will not re-fire (by design), but this could cause stale-closure bugs in tests. Not a blocker — behavior matches spec intent.
