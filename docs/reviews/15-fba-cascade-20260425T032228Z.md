## Issues Found

### Blocker

**`routes/cascades.py:47-49` — Unknown `adjustment_type` returns HTTP 400, not 422 as spec requires**

The spec acceptance criterion is explicit:
> `POST /apply-adjustment` with `adjustment_type="unsupported"` returns HTTP 422.

The implementation raises HTTP 400, and the test was subsequently changed to assert 400 instead of 422 (with a comment acknowledging the spec discrepancy). Changing the test to match an incorrect implementation is not an acceptable resolution — the AC is not satisfied.

The fix is a one-line change in `routes/cascades.py`:

```python
# Change:
raise HTTPException(status_code=400, detail=...)
# To:
raise HTTPException(status_code=422, detail=...)
```

The implementer noted this would also require updating `test_routes_cascades.py::test_unknown_adjustment_type_returns_400` (for swayback). That test should be updated to expect 422 as well, since the same route handles all adjustment types. Both the swayback spec (if it has this AC) and the route behavior should be consistent.

The test in `test_fba_cascade.py` at line ~367 (`test_unsupported_adjustment_type_returns_400`) must then be updated to assert 422.

---

### Important

**No test for missing pattern elements → HTTP 500**

The spec defines this error path:
> Pattern is missing required elements → HTTP 500 with detail naming the missing element

The route in `cascades.py` handles `ElementNotFound` and returns HTTP 500 with the element name. However, no test in `test_fba_cascade.py` covers this path. It is also not covered by `test_routes_cascades.py` (which predates FBA).

Suggested test: construct a minimal Pattern missing `front-side-panel` (or mock `apply_fba` to raise `ElementNotFound`) and assert HTTP 500 with the element name in the response detail.

---

### Nit

**`fba.py` — narration strings hardcoded inline; swayback uses `load_narration` from prompt files**

The two cascade implementations are inconsistent. `swayback.py` loads narration from `prompts/swayback/v1_baseline.md` via `load_narration()`. `fba.py` inlines all narration strings directly in the function body.

The spec says "Do not hardcode narration text as a module-level constant — put the strings inline with the step construction so they're easy to tune later," which permits inlining. However, CLAUDE.md's critical rule says "All prompts are files, not strings. Load from `prompts/` at runtime."

Whether narrative text counts as a "prompt" under CLAUDE.md's rule is ambiguous (the rule was likely targeting LLM prompt templates, not user-facing copy). Regardless, the codebase is now inconsistent and a `prompts/fba/` directory should be created if prompt-file loading is canonical. See Questions for Steph.

## Questions for Steph

1. **Route path:** The spec's acceptance criteria show `POST /apply-adjustment` (no prefix), but the route lives at `POST /cascades/apply-adjustment`. Integration tests use the prefixed path and pass. Is the `/cascades` prefix intentional? If the frontend spec (spec 11) will call `/apply-adjustment`, this needs to be resolved before frontend integration.

2. **Prompt-file convention for narration:** `swayback.py` loads narration via `load_narration()` from `prompts/swayback/v1_baseline.md`. `fba.py` inlines narration strings. The spec permits inlining, but CLAUDE.md says prompts are files. Should `prompts/fba/v1_baseline.md` be created and `fba.py` updated to use `load_narration("fba")`? Or is inline acceptable for non-LLM copy? Whichever is canonical should be applied consistently.

3. **422 vs 400 for unknown adjustment type (related to Blocker):** Once the 400→422 fix is made in `routes/cascades.py`, the existing swayback test `test_unknown_adjustment_type_returns_400` will fail. Should that test be updated to expect 422 as part of this fix, or should it be a separate cleanup?
