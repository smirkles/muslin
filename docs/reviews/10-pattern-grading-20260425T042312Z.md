## Issues Found

### Blocker: None

### Important: Out-of-scope dependency `replicate>=1.0.7` added to `pyproject.toml`

**File:** `backend/pyproject.toml` and `backend/uv.lock`

The spec states explicitly: "External libraries needed: none (stdlib `uuid`, `dataclasses`)." The `replicate` package was pulled in during an intermediate commit (`6b7311e`) that added failing tests for spec 12 (SAM2 segmentation). That test commit was correctly cleaned up (`68758c3` removed the test files), but the dependency addition to `pyproject.toml` and `uv.lock` was not reversed.

`replicate` is not imported anywhere in `lib/`, `routes/`, or the tests on this branch. This is dead weight in the production dependency list and is squarely out of scope for spec 10.

**Fix:** Remove `"replicate>=1.0.7"` from `backend/pyproject.toml` dependencies, then run `uv lock` to regenerate `uv.lock`.

### Important: `lib/grading.py` imports private internals of `pattern_ops`

**Files:** `backend/lib/grading.py` lines 26, 117, 119, 122, 177, 182

`grading.py` directly uses:
- `_element_bbox` (private function, line 26 import and line 177 usage)
- `_G_TAGS` (private module-level set, imported inside `_piece_ids_from_pattern` at line 117)
- `pattern._tree.getroot()` (private attribute, line 119)
- `pattern._id_index` (private attribute, lines 182)

CLAUDE.md rule 2 says "All SVG manipulation goes through `backend/lib/pattern_ops/`." The private attributes and helpers are implementation internals of `pattern_ops`. Grading reaching into them directly creates a hidden coupling: any rename or refactor of `pattern_ops` internals breaks `grading.py` silently.

The fix for `_element_bbox` and piece enumeration is to make them part of `pattern_ops`'s public API. The ADR already filed (`003-scale-element.md`) explicitly notes `_element_bbox` is useful for future operations — it should have been surfaced as a public function at the same time. Exposing `piece_ids(pattern)` and `element_bbox(pattern, element_id)` as public functions in `pattern_ops.py` would let `grading.py` avoid all private access.

**Fix:** In `pattern_ops.py`:
- Rename `_element_bbox` to `element_bbox` (public).
- Add a public `piece_ids(pattern: Pattern) -> list[str]` function that encapsulates the `_G_TAGS` / `_tree.getroot()` / `_id_index` access.

Then update `grading.py` to call these public functions.

### Important: `user_measurements: object` is a weak type hint on a public function

**File:** `backend/lib/grading.py` line 132

CLAUDE.md requires "Type hints on all public functions." The parameter type `object` technically satisfies the requirement but defeats it — the actual contract is structural: the argument must have `bust_cm`, `waist_cm`, `hip_cm`, and `back_length_cm` float attributes.

A `typing.Protocol` would express this contract clearly, catch misuse at type-check time, and serve as documentation:

```python
from typing import Protocol

class MeasurementsProtocol(Protocol):
    bust_cm: float
    waist_cm: float
    hip_cm: float
    back_length_cm: float
```

**Fix:** Define `MeasurementsProtocol` in `grading.py` and change the signature to `user_measurements: MeasurementsProtocol`.

### Nit: Four `# type: ignore[arg-type]` in `routes/patterns.py`

**File:** `backend/routes/patterns.py` lines 127–130

After the `if any(v is None ...)` guard block, mypy cannot narrow the type from `float | None` to `float`. The `# type: ignore[arg-type]` comments suppress the error rather than resolving it. A `cast` from `typing` or narrow `assert` would make this explicit without suppression.

```python
from typing import cast
base_meas = BaseMeasurements(
    bust_cm=cast(float, meta.base_bust_cm),
    ...
)
```

CLAUDE.md flags suppressed warnings without justification as a review item.

## Test Quality Assessment

Overall quality is high. Tests are organized into clearly named classes matching the acceptance criteria, fixtures are reusable, and assertions are specific.

One gap relative to the spec:

- **Acceptance criterion #2** specifies "every SVG coordinate matches the input within 1e-6." The tests verify bounding-box width and height are unchanged (`test_identity_bodice_width_unchanged`, `test_identity_skirt_height_unchanged`), but not absolute coordinate positions. A uniform piece drift of `(+5, +5)` while preserving dimensions would pass these tests. This matters less because the pivot math for sx=sy=1.0 is trivially exact, but a point-level coordinate comparison against the original SVG string would make this criterion unambiguous. Low severity — acceptable for merge if other issues are fixed, but note for future test hygiene.

- No test covers a piece with an unrecognised id prefix (the `sx=1.0` warning branch). The spec doesn't require this case to be tested explicitly, but it would be good defensive coverage.
